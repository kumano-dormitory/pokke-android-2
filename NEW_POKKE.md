# POKKE v2 機能仕様書

> 熊野寮 荷物管理アプリ（Android タブレット）

---

## 概要

POKKEは熊野寮の荷物管理アプリ。一台のタブレット（Android）で寮生に対する荷物の受付・登録・管理を行う。
**Android側も自前でDBを持ち**、オフライン時もローカルに保存、ネット復帰後にサーバーへ同期する。

### 利用シーン

- 荷物が届いたとき、事務室のタブレットで荷物を登録する
- 寮生が荷物を取りに来たとき、受け渡しを記録する
- 荷物が届いたことをDiscordで寮生に通知する

---

## 主体（Actor）

| 主体 | 説明 | 操作内容 |
|------|------|----------|
| **事務当番** | タブレットを操作する人 | 荷物の登録、引き渡し、泊まり事務当番、寮生呼び出し、操作履歴の閲覧 |
| **寮生** | 荷物の受け取り主（アプリ操作は行わない） | Discordで通知を受ける |
| **管理者** | Webアプリから設定を行う | サーバー同期管理、荷物紛失反映 |

### 5. 泊まり事務当番

| 項目 | 内容 |
|------|------|
| **主体** | 事務当番 |
| **説明** | 未受け渡しの荷物が全て棚にあるかを確認する |
| **UI** | 旧UIを踏襲 |

#### フロー（旧アプリと同様の2フェーズ）

**フェーズ1: 現物確認**
- 未引渡荷物を棟別（A棟/B棟/C棟/臨キャパ）にリスト表示
- 各荷物についてチェックボックスで確認
- 見つからない荷物は「紛失」トグル
- 全件チェック完了でフェーズ2へ

**フェーズ2: 荷物札確認**
- 同じリストだが紛失トグルは無効（表示のみ）
- 全件チェック完了で泊まり事務当番終了

#### 記録内容

- 各荷物の `lastConfirmedAt` を更新
- 操作ログに `NIGHT_DUTY_CONFIRM` を記録

## データモデル

### 外部キー制約方針

寮生データはサーバーから全件再同期（DELETE + INSERT）する場合がある。
その際に関連データが壊れないよう、以下の外部キー制約を適用する。

| 参照元テーブル | 参照先 | ON DELETE | 理由 |
|---------------|--------|-----------|------|
| **Parcel** → Ryosei | `ryoseiId` | **SET NULL** | 寮生が再同期で消えても荷物データは残す。冗長保持のownerName等で識別可能 |
| **ParcelOperationLog** → Parcel | `parcelId` | **SET NULL** | 荷物が物理削除されても操作履歴は残す |
| **DutyPerson** → Ryosei | `ryoseiId` | **CASCADE** | 寮生が消えたら事務当番も解除（再設定が必要） |

### Ryosei（寮生キャッシュ — Android側）

```
Ryosei {
  id              String         // 寮生ID
  name            String         // 名前
  nameKana        String         // 名前（かな）
  nameAlphabet    String         // 名前（ローマ字）
  room            String         // 部屋名（例: A301）
  block           String         // ブロック（例: A1, B12, C34）
  leavingDate     LocalDate?     // 退寮日（在寮中はnull）
  discordStatus   DiscordStatus  // Discord紐づけ状態
}

enum DiscordStatus {
  LINKED     // 紐づけ済
  UNLINKED   // 未紐づけ
}
```

### Parcel（荷物）

```
Parcel {
  id                String        // ULID
  createdAt         DateTime      // 作成日時
  updatedAt         DateTime      // 更新日時

  // 受け取り主
  ryoseiId          String?       // 寮生ID（FK → Ryosei, ON DELETE SET NULL）
  ownerBlock        String        // ブロック（冗長保持）
  ownerRoomName     String        // 部屋名（冗長保持）
  ownerName         String        // 寮生名（冗長保持）

  // 荷物情報
  parcelType        ParcelType    // 荷物種別
  note              String?       // 備考

  // 状態
  status            ParcelStatus  // REGISTERED / RECEIVED
  isLost            Boolean       // 紛失フラグ

  // 登録情報
  registeredByName  String        // 登録した事務当番名

  // 引渡情報
  deliveredAt       DateTime?     // 引渡日時
  deliveredByName   String?       // 引渡した事務当番名

  // 同期
  syncedAt          DateTime?     // サーバー同期完了日時
  deviceId          String?       // デバイス識別子

  // 泊まり事務当番
  lastConfirmedAt   DateTime?     // 最終確認日時
}

enum ParcelType {
  NORMAL          // 普通
  REFRIGERATED    // 冷蔵
  FROZEN          // 冷凍
  LARGE           // 大型
  ABSENCE_SLIP    // 不在票
  OTHER           // その他
}

enum ParcelStatus {
  REGISTERED      // 登録済み（未受け取り）
  RECEIVED        // 受け渡し済み
}
```

### DutyPerson（事務当番）

```
DutyPerson {
  id        String    // ULID
  ryoseiId  String    // 寮生ID（FK → Ryosei, ON DELETE CASCADE）
  name      String    // 事務当番の名前（冗長保持）
  updatedAt DateTime  // 更新日時
}
```

※ 1レコードのみ。交代時に上書き。
※ ryoseiIdにCASCADE制約あり。寮生の全件再同期時にDutyPersonもクリアされるため、再設定が必要。

### ParcelOperationLog（操作履歴）

```
ParcelOperationLog {
  id              String                // ULID
  createdAt       DateTime              // 操作日時
  parcelId        String?               // 荷物ID（取消で物理削除後はnull）
  operationType   ParcelOperationType   // 操作種別
  operatedByName  String?               // 操作した事務当番名
  metadata        Json?                 // 補足情報
}

enum ParcelOperationType {
  REGISTER              // 荷物登録
  DELIVER               // 荷物引渡
  CANCEL_REGISTER       // 登録取消（荷物を物理削除）
  CANCEL_DELIVER        // 引渡取消（statusをREGISTEREDに戻す）
  MARK_LOST             // 紛失記録
  NIGHT_DUTY_CONFIRM    // 泊まり事務当番確認
  DUTY_CHANGE           // 事務当番交代
}
```

## 操作取消ルール

| 操作 | 取消可能条件 | 取消時の処理 |
|------|------------|------------|
| 荷物登録 | 登録から**10分以内** | 荷物を物理削除。操作ログのmetadataに荷物情報を退避 |
| 荷物引渡 | 引渡から**10分以内** | deliveredAt/deliveredByNameをクリア、statusをREGISTEREDに戻す |

## 設計原則: オフラインファースト

**全ての操作はローカルDB（Room/SQLite）で完結する。** サーバーはバックアップ・通知用であり、サーバーが落ちていてもアプリの全機能が使える。

### アーキテクチャ

```
[Androidタブレット — マスター]              [サーバー — バックアップ・通知]
┌──────────────────────┐                  ┌─────────────────────┐
│ ローカルDB (Room)     │                  │ PostgreSQL          │
│ ・Ryosei（マスター）  │ ←─差分取得─       │ ・Ryosei（マスター） │
│ ・Parcel（マスター）  │ ──バッチ同期──→   │ ・Parcel（バックアップ）│
│ ・DutyPerson（ローカル専用）│              │                     │
│ ・OperationLog（ローカル専用）│            │                     │
└──────────────────────┘                  └─────────────────────┘
```

### データの管理方針

| データ | マスター | サーバー同期 | 備考 |
|--------|----------|-------------|------|
| **寮生 (Ryosei)** | サーバー | 差分取得のみ | 普段の操作ではローカルDBから取得。定期的にサーバーから差分を取得して更新 |
| **荷物 (Parcel)** | ローカルDB | バッチで差分アップロード | 登録・引渡等は全てローカルで即時処理。サーバーへはバッチ処理で差分を反映（通信負荷軽減） |
| **事務当番 (DutyPerson)** | ローカルDB | **同期しない** | タブレット内でのみ管理。サーバーに送らない |
| **操作履歴 (OperationLog)** | ローカルDB | **同期しない** | タブレット内でのみ管理。サーバーに送らない |

### 同期の流れ

1. **全ての操作はローカルDBに即時保存**（サーバー通信は不要）
2. **寮生データ**: アプリ起動時 or 手動更新時にサーバーから差分取得（`updatedAt` ベース）
3. **荷物データ**: バックグラウンドで定期的にバッチ同期（`syncedAt = null` のレコードを一括送信）
4. **Discord通知**: 荷物データのバッチ同期完了後にサーバー側で送信

### API エンドポイント

#### 寮生

| メソッド | パス | 説明 |
|----------|------|------|
| GET | `/api/ryosei` | 寮生全件取得（初回同期・ゼロからの再同期用） |
| GET | `/api/ryosei?updated_after={datetime}` | 指定日時以降に更新された寮生のみ取得（差分同期用） |

#### 荷物（バッチ同期のみ）

| メソッド | パス | 説明 |
|----------|------|------|
| POST | `/api/parcels/sync` | 未同期の荷物操作を一括アップロード（登録・引渡・取消・紛失を含む） |

※ 個別の登録・引渡・取消APIは不要。全てバッチ同期エンドポイントに集約する。

## Discord通知

### 荷物到着通知（自動）

荷物登録がサーバーに同期された際、受取人の寮生にDiscord DMを送信する。

**送信条件**: 寮生の `discordStatus = LINKED`

**メッセージ例**:
```
📦 荷物のお知らせ

○○ さん宛の荷物が届きました。
事務室までお越しください。

登録日時: 2026/03/17 14:30
荷物の種類: 普通
```
